## 3. Build & SSR Setup

### `next.config.ts` â€” Zero-runtime extraction

Use `withTastyZero` to enable the Babel plugin that extracts `tastyStatic()` calls into a static CSS file at build time:

```ts
import { withTastyZero } from '@tenphi/tasty/next';

export default withTastyZero({
  output: 'public/tasty.css',
})({
  reactStrictMode: true,
});
```

The generated `public/tasty.css` is imported in the root layout so global styles load without runtime JS.

### `app/tasty-registry.tsx` â€” SSR for runtime styles

```tsx
'use client';

import './tasty-config';

import { TastyRegistry } from '@tenphi/tasty/ssr/next';

export default function TastyStyleRegistry({
  children,
}: {
  children: React.ReactNode;
}) {
  return <TastyRegistry>{children}</TastyRegistry>;
}
```

### `app/layout.tsx`

```tsx
import './global-styles';
import '@/public/tasty.css';
import TastyStyleRegistry from './tasty-registry';

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <TastyStyleRegistry>{children}</TastyStyleRegistry>
      </body>
    </html>
  );
}
```

### Configuration Loading

`app/tasty-config.ts` is a `'use client'` module that calls `configure()` with tokens, presets, and states. It is imported at the top of the registry component so configuration is set before any `tasty()` component renders.

## 4. Color System (Glaze)

### `app/theme.ts`

Define the color palette using Glaze. Each color role is a separate Glaze theme. They share the same color structure but differ in hue.

```ts
import { glaze } from '@tenphi/glaze';

// Primary theme â€” purple hue
const primary = glaze(272, 75);

primary.colors({
  surface:       { lightness: 98, saturation: 0.5 },
  'surface-2':   { lightness: 95, saturation: 0.6 },
  'surface-3':   { lightness: 91, saturation: 0.7 },
  text:          { base: 'surface', lightness: '-62', contrast: 'AAA' },
  'text-soft':   { base: 'surface', lightness: '-40', contrast: 'AA' },
  border:        { base: 'surface', lightness: ['-12', '-25'], contrast: 'AA-large' },
  'accent-fill': { lightness: 52, mode: 'fixed' },
  'accent-text': { base: 'accent-fill', lightness: '+48', contrast: 'AA', mode: 'fixed' },
  icon:          { lightness: 55, saturation: 0.9 },
  disabled:      { lightness: 80, saturation: 0.3 },
  'shadow-sm':   { type: 'shadow', bg: 'surface', fg: 'text', intensity: 5 },
  'shadow-md':   { type: 'shadow', bg: 'surface', fg: 'text', intensity: 12 },
  'shadow-lg':   { type: 'shadow', bg: 'surface', fg: 'text', intensity: 22 },
  overlay:       { lightness: 10, opacity: 0.5 },
});

// Derived role themes
const danger  = primary.extend({ hue: 15 });
const success = primary.extend({ hue: 155 });
const warning = primary.extend({ hue: 70 });
const note    = primary.extend({ hue: 210 });

// Compose palette
const palette = glaze.palette({ primary, danger, success, warning, note });

// Export as Tasty-compatible style-to-state bindings
// Output shape: { '#primary-surface': { '': 'oklch(...)', '@dark': 'oklch(...)' }, ... }
export const colorTokens = palette.tasty({ prefix: true, format: 'oklch' });
```

### Color Token Naming Convention

Tokens are auto-prefixed by theme name. Usage in styles:

- `#primary-surface`, `#primary-text`, `#primary-accent-fill`
- `#danger-surface`, `#danger-text`, `#danger-accent-fill`
- `#success-text`, `#warning-border`, `#note-icon`

### Registering Tokens via Recipe

The Glaze `.tasty()` output is a style object with color token keys (`#name`) mapped to state objects (`{ '': value, '@dark': value }`). These define CSS custom properties on whichever element they're applied to. The correct integration is to register them as a **recipe** and then apply that recipe on a root element (e.g. `body`).

In `tasty-config.ts`:

```ts
import { colorTokens } from './theme';

configure({
  recipes: {
    palette: colorTokens,
  },
});
```

Then in `global-styles.ts`, apply the recipe on `body`:

```ts
tastyStatic('body', {
  recipe: 'palette',
  fill: '#primary-surface',
  color: '#primary-text',
  font: true,
  preset: 't2',
});
```

This generates CSS custom properties (`--primary-surface-color`, `--primary-text-color`, etc.) on `body` with appropriate `@dark` state selectors. All descendant components can then reference `#primary-surface`, `#primary-text`, etc. as color tokens.

## 6. Reusable Primitives (`app/ui/`)

All primitives live in `app/ui/`, one component per file. Every primitive is `'use client'` and uses `tasty()`.

**Style prop groups exposed on primitives:**

| Group | Contains | Exposed On |
|---|---|---|
| `BASE_STYLES` | display, font, preset, hide, whiteSpace, opacity, transition | All primitives |
| `OUTER_STYLES` | `POSITION_STYLES` + `DIMENSION_STYLES` + `BLOCK_OUTER_STYLES` (margin, placeSelf, width, height, border, radius, shadow, â€¦) | All primitives |
| `BLOCK_STYLES` | padding, overflow, scrollbar, textAlign + border, radius, shadow, outline | All block-level primitives (Space, Grid, Button, Link, Badge, CodeBlock) |
| `FLOW_STYLES` | flow, gap, align, justify, placeItems, placeContent, gridColumns, gridRows, â€¦ | Layout primitives (Space, Grid) |

Every primitive exposes `BASE_STYLES` + `OUTER_STYLES` at minimum, so visibility, typography, transitions, position, and dimensions can always be controlled from the outside. Block-level elements additionally expose `BLOCK_STYLES` for padding/overflow control.

### 6.7 `Switcher` â€” Segmented toggle (`app/ui/Switcher.tsx`)

A generic segmented control that renders a row of options, one of which is active. Accepts `options`, `value`, and `onChange` props. The styling is fully reusable â€” no domain logic baked in.

```tsx
const SwitcherElement = tasty({
  styles: {
    display: 'inline-flex',
    flow: 'row',
    gap: 0,
    padding: '0.25x',
    radius: 'round',
    fill: '#primary-surface-2',
    border: true,
  },
  styleProps: [...BASE_STYLES, ...OUTER_STYLES, ...BLOCK_STYLES],
});

const SwitcherOption = tasty({
  as: 'button',
  styles: {
    display: 'inline-flex',
    placeItems: 'center',
    padding: '0.5x 1.5x',
    preset: 't4',
    radius: 'round',
    cursor: 'pointer',
    transition: 'theme',
    fill: {
      '': 'transparent',
      'active': '#primary-surface',
    },
    color: {
      '': '#primary-text-soft',
      'active': '#primary-text',
    },
    shadow: {
      '': 'none',
      'active': '0 1x 2x #primary-shadow-sm',
    },
    border: 'none',
  },
});
```

The `Switcher` React component maps `options` to `SwitcherOption` buttons, setting the `active` modifier on the one matching `value`:

```tsx
interface SwitcherProps {
  options: { value: string; label: string }[];
  value: string;
  onChange: (value: string) => void;
}

function Switcher({ options, value, onChange, ...props }: SwitcherProps) {
  return (
    <SwitcherElement {...props}>
      {options.map((opt) => (
        <SwitcherOption
          key={opt.value}
          mods={{ active: opt.value === value }}
          onPress={() => onChange(opt.value)}
        >
          {opt.label}
        </SwitcherOption>
      ))}
    </SwitcherElement>
  );
}
```

### 9.1.1 Theme Switcher (`app/components/ThemeSwitcher.tsx`)

A three-way toggle that sets the color scheme: **Light**, **Dark**, **System**.

**How it works with the `@dark` state:**

The `@dark` state is declared as:

```
@root(schema=dark) | (!@root(schema) & @media(prefers-color-scheme: dark))
```

This means:
- `html[data-schema="dark"]` â†’ dark mode (explicit override)
- `html[data-schema="light"]` â†’ light mode (explicit override, because `@root(schema)` is truthy but not `dark`)
- No `data-schema` attribute â†’ follows OS preference (`prefers-color-scheme`)

**Switcher behavior:**

| Selection | Action |
|---|---|
| Light | `document.documentElement.dataset.schema = 'light'` |
| Dark | `document.documentElement.dataset.schema = 'dark'` |
| System | `delete document.documentElement.dataset.schema` |

The selection is persisted in `localStorage` and restored on page load via an inline `<script>` in the root layout (to avoid flash).

**Implementation:** `ThemeSwitcher` is a thin behavioral wrapper around the generic `Switcher` primitive from `app/ui/Switcher.tsx`. No custom styling â€” just wiring up the options and the `onChange` handler:

```tsx
import { Switcher } from '@/app/ui/Switcher';

const THEME_OPTIONS = [
  { value: 'light', label: 'Light' },
  { value: 'dark', label: 'Dark' },
  { value: 'system', label: 'System' },
];

function ThemeSwitcher() {
  const [theme, setTheme] = useState(() => localStorage.getItem('theme') ?? 'system');

  function handleChange(value: string) {
    setTheme(value);
    localStorage.setItem('theme', value);

    if (value === 'system') {
      delete document.documentElement.dataset.schema;
    } else {
      document.documentElement.dataset.schema = value;
    }
  }

  return <Switcher options={THEME_OPTIONS} value={theme} onChange={handleChange} />;
}
```

**Feature list** (derived from README "Why Tasty"):

## 10. Responsive Design

Use Tasty predefined states throughout:

| State | Breakpoint | Usage |
|---|---|---|
| `@mobile` | `< 768px` | Single-column layouts, smaller typography, reduced padding |
| `@tablet` | `< 1024px` | 2-column grids instead of 3, adjusted spacing |
| `@desktop` | `>= 1024px` | Full 3-column layouts, maximum content width |

### Responsive Patterns

Apply via state maps on style properties:

```tsx
padding: {
  '': '12x 4x',
  '@tablet': '8x 3x',
  '@mobile': '6x 2x',
}

gridColumns: {
  '': '1sf 1sf 1sf',
  '@tablet': '1sf 1sf',
  '@mobile': '1sf',
}

preset: {
  '': 'd1',
  '@mobile': 'h1',
}

flow: {
  '': 'row',
  '@mobile': 'column',
}
```

## 11. Visual Design Guidelines

### Typography

- **Primary font:** Tasty default system font stack (system-ui, -apple-system, sans-serif via `$font`).
- **Monospace font:** Tasty default monospace stack (ui-monospace, monospace via `$monospace-font`).
- No custom fonts loaded â€” system fonts only for fast, zero-request typography.
- Use `preset` for all typography. Never set `fontSize` directly.

### Colors

- Primary palette is purple-based (`hue: 272`).
- Surfaces use very high lightness (95â€“98) with low saturation for subtle color.
- Text uses WCAG AAA contrast against surfaces.
- Accent fill uses `mode: 'fixed'` so brand purple stays recognizable in dark mode.
- All colors auto-adapt to dark scheme via Glaze.

### Spacing

- Base gap: `8px` (`1x`).
- Component padding typically `2x`â€“`4x`.
- Section padding: `12x` vertical, `4x` horizontal (desktop).
- Consistent gap between elements: `2x` (tight), `4x` (normal), `8x` (loose).

### Borders & Radius

- Default border: `1bw solid #border` (1px solid, color from palette).
- Card radius: `1cr` (16px).
- Button radius: `round` (pill shape).
- Small element radius: `1r` (8px).

### Shadows

- Use Glaze shadow tokens: `#primary-shadow-sm`, `#primary-shadow-md`, `#primary-shadow-lg`.
- Apply via `shadow` property: `shadow: '0 1x 3x #primary-shadow-sm, 0 2x 8x #primary-shadow-md'`.

### Transitions

- Use semantic transition names: `theme`, `fill`, `shadow`.
- Default timing: `0.2s` (`$transition` token).
- Respect `@reduce-motion` state for users who prefer reduced motion.

### Dark Mode

- Controlled via `@dark` predefined state and Glaze token adaptation.
- Three-way theme switcher in the header: Light / Dark / System.
- Sets `html[data-schema]` attribute (`'light'`, `'dark'`, or removed for system).
- All Glaze tokens include `@dark` state variants automatically.
- Selection persisted in `localStorage`, restored via inline `<script>` to prevent FOUC.

## 12. Code Block Styling

Code examples on the landing page are pre-formatted `<pre><code>` elements (not a runtime syntax highlighter). Simple color spans for keywords, strings, and comments.

```tsx
const CodeBlock = tasty({
  as: 'pre',
  styles: {
    padding: '3x',
    fill: '#primary-text',
    color: '#primary-surface',
    radius: '1cr',
    overflow: 'auto',
    preset: 'c2',
    font: 'monospace',
    border: 'none',
    scrollbar: 'thin',
    margin: 0,
    Code: {
      $: '>',
      font: 'monospace',
      preset: 'c2',
      whiteSpace: 'pre',
    },
  },
  elements: {
    Code: 'code',
  },
  styleProps: [...BASE_STYLES, ...OUTER_STYLES, ...BLOCK_STYLES],
});
```

Syntax highlight colors defined as inline styles or additional `<span>` styled components with fixed colors for simplicity (no external syntax highlighter dependency).

## 14. Performance Considerations

- **SSR first:** All runtime styles are server-rendered via `TastyRegistry`. No FOUC.
- **Static globals:** Global tokens and body styles are extracted at build time via `tastyStatic` into `public/tasty.css` â€” zero runtime cost.
- **System fonts:** No custom font files to download. Zero font-related network requests.
- **Minimal JS:** No client-side syntax highlighting library. Code blocks are static HTML.
- **Image-free:** No raster images. Icons are emoji or inline SVG. Logo is text-based.
- **Theme persistence:** Inline `<script>` in layout restores `data-schema` before paint to prevent dark mode flash.

## 16. Conventions & Best Practices

1. **Always use `tasty()` wrappers** â€” never pass `styles` prop directly to components.
2. **Use `preset` for typography** â€” never set `fontSize` or `lineHeight` directly.
3. **Use `fill` instead of `background`** â€” Tasty's recommended prop.
4. **Use `flow` instead of `flexDirection`** â€” Tasty's unified direction control.
5. **Use color tokens everywhere** â€” `#text`, `#surface`, not raw color values.
6. **Use custom units** â€” `2x` for spacing, `1r` for radius, `1bw` for borders.
7. **Use state maps for responsive** â€” `{ '': 'value', '@mobile': 'value' }`.
8. **Use `elements` for sub-components** â€” typed sub-components via `Section.Title`.
9. **Use `$` affix on sub-elements** â€” `$: '>'` for direct children to prevent style leakage.
10. **Expose `BASE_STYLES` + `OUTER_STYLES` on all primitives** â€” so visibility, typography, transitions, position, size, and margin can be set from outside. Add `BLOCK_STYLES` for block-level elements.
11. **Use semantic `transition` names** â€” `theme`, `fill`, `shadow`, not raw CSS properties.
12. **Use `tastyStatic` for global/static styles** â€” body tokens, :root colors. Use `tasty()` for interactive components.
13. **One primitive per file in `app/ui/`** â€” compound components in `app/components/`.
14. **All components are `'use client'`** â€” since `tasty()` uses hooks internally. Pages can be server components that import client components.
